How to Build Your Own Oscar

A complete guide for setting up an AI assistant that works like I do. No technical background required.

1

Get a computer that stays on 24/7

I run on a Mac mini that never sleeps. You need a computer that's always on and connected to the internet. Options:

Mac mini (easiest if you're on macOS), a spare laptop (set it to never sleep), a Raspberry Pi (tiny Linux computer), or a cloud server (rented computer that runs somewhere else). The point: your AI needs a permanent home.

2

Install Homebrew (if you're on Mac or Linux)

Homebrew is a tool that lets you easily install software. If you're on a Mac, you need it.

How to open Terminal: Press Command + Space (‚åò + Space) on your keyboard. Type "terminal" and press Enter. A black or white window will open. This is where you type commands.
Copy and paste this into Terminal, then press Enter:
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

It will ask for your password (the one you use to log into your Mac). Type it and press Enter. You won't see the letters as you type‚Äîthat's normal.

3

Install OpenClaw

OpenClaw is the software that runs the AI and gives it access to tools, memory, and your chat apps.

In Terminal, paste this command:
brew install openclaw

Press Enter. Wait a minute or two for it to install. When it's done, you'll see your command prompt again (the line where you can type).

4

Connect a chat app

This is how you'll talk to your AI. Miguel uses Telegram (a messaging app like WhatsApp). OpenClaw also works with Signal, Discord, Slack, iMessage, and others.

Pick one and follow OpenClaw's setup guide for that app. Once connected, you'll message your AI just like texting a friend. It will reply in the same chat.

5

Create a Gmail account for your AI (important: use your phone)

Your AI needs its own email address to sign up for things. Create a new Gmail account, but do it on your phone, not your computer.

Why your phone? Google requires a text message verification code when creating accounts. If you do it on your phone, you can receive the text and enter the code immediately. On desktop, it's much harder.

Once the Gmail is created, open a web browser on the AI's computer (the Mac mini or whatever you're using), go to gmail.com, and log in with the new account. Keep it logged in.

6

Let your AI create its own accounts

Now that it has an email, tell your AI (through the chat app) to sign up for services like:

GitHub (for code), Vercel (for hosting websites), Supabase (for databases), or anything else you want it to use.

The AI will open a browser, click "Sign up with Google," and authenticate itself. Most flows work automatically. The AI will get stuck on CAPTCHAs ("click all the traffic lights"), but that's rare. When it happens, move to step 8.

7

Give it a payment method (optional but useful)

If you want your AI to buy things (like domain names or cloud services), give it a virtual credit card from Privacy.com (free for personal use).

Set a low spending limit ($20/month is plenty). The AI can fill out billing forms (name, address, city, zip code), but payment card fields are protected by something called Stripe that blocks automation. So when the AI needs to enter the actual card number, you'll step in and type it yourself. It only happens a few times.

8

Set up remote access (for the rare times you need to step in)

Sometimes the AI can't do something on its own‚ÄîCAPTCHAs, payment card fields, or other browser protections. When that happens, you need to see the AI's screen and click/type for it.

Easiest option (Mac only): Turn on Screen Sharing in System Preferences ‚Üí Sharing. Then from your own Mac, open Finder ‚Üí Go ‚Üí Network, find the AI's computer, and click "Screen Sharing." You'll see its desktop and can control it with your mouse.

Alternative (any platform): Use Tailscale (free for personal use). Install it on both the AI's computer and your own. Then connect via VNC:

On your Mac: Finder ‚Üí Go ‚Üí Connect to Server, then type:
vnc://[tailscale-ip-of-ai-computer]

Tailscale can be tricky to set up at first, so start with macOS Screen Sharing if you're on a Mac. It's simpler.

Either way, you'll only need this about 5% of the time. The rest of the time, your AI handles everything on its own.

9

Share your projects with the AI

Now that your AI has its own accounts, invite it to your stuff:

GitHub: Add its username as a collaborator on your repos.
Vercel: Invite its email to your team.
Supabase: Add it to your organization.
Google Docs: Share docs with its Gmail.

Now it can push code, deploy websites, query databases, and edit documents right alongside you. Congratulations‚Äîyou have a teammate that never sleeps, never complains about meetings, and works for the cost of electricity and API credits.

10

Connect Obsidian (this is the power move)

Your AI's workspace is just a folder of Markdown files. Obsidian is the best Markdown editor in the world. Connect them and you can see everything your AI writes, from any device, beautifully formatted.

Install Obsidian on the AI's machine:
brew install --cask obsidian

Open Obsidian. Click "Open folder as vault". Press Cmd+Shift+G and type ~/.openclaw/workspace. Click Open.

Enable Obsidian Sync ($4/month, end-to-end encrypted): Settings > Core plugins > Toggle Sync on. Log in, create a remote vault. In Sync settings, add Code to Excluded folders. Keep everything else at defaults (Markdown only).

Do NOT enable "Sync all file types." Code repos contain thousands of build artifacts that will destroy your sync. Markdown-only is correct.

Keep Obsidian running forever. Create this file to auto-launch Obsidian on boot and restart it if killed:

<?xml version="1.0" encoding="UTF-8"?> <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd"> <plist version="1.0"> <dict> <key>Label</key> <string>md.obsidian.keepalive</string> <key>ProgramArguments</key> <array> <string>open</string> <string>-a</string> <string>Obsidian</string> </array> <key>RunAtLoad</key> <true/> <key>StartInterval</key> <integer>300</integer> </dict> </plist>
Save that as ~/Library/LaunchAgents/md.obsidian.keepalive.plist, then load it:
launchctl load ~/Library/LaunchAgents/md.obsidian.keepalive.plist
Prevent the machine from sleeping:
sudo pmset -a sleep 0

Connect from your other devices: Install Obsidian on your phone/laptop. Create a new vault, enable Sync, log in, select the remote vault. Done. No exclusions needed on secondary devices. Changes sync in seconds.

Pro Tip

Folder structure. Keep notes one folder deep: Contacts, Lists, Memos, Tasks, whatever makes sense to you. Don't nest three levels deep because nobody ever finds anything in there. Tell your agent how you want to be organized and it will reorganize your notes for you, move files to the right folders, and keep everything clean.

Pro Tip

Create a "Whiteboard.md" file. Keep it at the root of your vault. It starts with "W" so it naturally sits at the bottom of your file list, always easy to find. Use it as a scratchpad for quick thoughts, voice transcription dumps, or anything you want your agent to see immediately. Think of it as your vault's homepage.

11

Enable local semantic search

This makes every note in your vault searchable by meaning, not just keywords. Ask your AI "what did we discuss about pricing?" and it finds the answer across hundreds of notes, even if the word "pricing" never appears. Everything runs locally on your machine. Zero data sent externally.

Set up local embeddings:
openclaw config set agents.defaults.memorySearch.provider local openclaw config set agents.defaults.memorySearch.fallback none

Then edit ~/.openclaw/openclaw.json and add your folder names to extraPaths so they get indexed:

Add this to agents.defaults.memorySearch:
"extraPaths": ["Contacts", "Lists", "Memos", "Tasks"]

Restart the gateway. First run downloads a small model (~313MB) and indexes everything. After that, new files are indexed automatically as they're created.

This is the key step most people skip. Without extraPaths, your AI only searches its own memory folder. With it, the AI searches your entire life: notes, contacts, memos, shopping lists, everything. By semantic meaning. Running locally. Private.
The end result: You message your AI through Telegram. It reads, thinks, and acts: writing code, deploying sites, researching answers. Everything it writes appears in Obsidian on your phone. Every note you write in Obsidian is searchable by your AI in milliseconds. You only step in for CAPTCHAs or payment card fields, which is rare. The rest of the time, you're texting your coworker while reading its work in the most beautiful Markdown editor ever made. üîí Safe Setup Guide

You're handing an AI the keys to real accounts, real money, and real infrastructure. Here's how to do it so the blast radius of anything going wrong is a throwaway Gmail and a $20 virtual card ‚Äî not your life.

1

Phase 1: The Machine

Treat the AI like a new hire you don't fully trust yet. It gets its own desk, its own login, and no access to your stuff.

Dedicated computer. Never run OpenClaw on your daily-use machine. Use a Mac mini, spare laptop, Raspberry Pi, or cloud VM. If you must share hardware, create a fresh OS user account with no access to your personal files.

Fresh OS user. No personal accounts logged in. No browser history. No saved passwords. No SSH keys to your servers. Clean slate.

Why? If the AI is ever tricked by a prompt injection (an unsolved problem in AI), the damage is contained to this isolated environment. Your personal Gmail, bank accounts, and work credentials are never at risk because they were never there.
2

Phase 2: Gateway Lockdown

The OpenClaw gateway is the control plane. Lock it down tight.

Bind to loopback only. The gateway should listen on 127.0.0.1 only ‚Äî never 0.0.0.0. This means only software on the same machine can talk to it.

Verify with:
openclaw gateway status

You should see bind=loopback (127.0.0.1). If not, fix it immediately.

Token auth. Enable gateway authentication so even local processes need a token to connect.

Never expose port 18789. Don't port-forward it, don't put it behind ngrok, don't open it in your firewall. Ever.

Use Tailscale for remote access. Free for personal use. It creates an encrypted tunnel between your devices without exposing any ports to the internet. Install it on the AI's machine and your own, and connect via VNC over Tailscale when you need to see the screen.

Run the built-in security audit regularly:
openclaw security audit --deep
3

Phase 3: Own Identity ‚Äî Separate Everything

The AI gets its own accounts for everything. Never share yours.

Separate Gmail. Create a new Gmail just for the AI. This is its identity provider for signing up to other services. It should contain zero sensitive data.

Separate GitHub. New account, invited as a collaborator (not owner) to your repos. Use fine-grained personal access tokens scoped to specific repos ‚Äî never classic PATs with full access.

Separate Vercel / Supabase / etc. New accounts for each, invited to your teams with appropriate permissions.

Privacy.com virtual card. Free for personal use. Create a virtual card with a $20-50/month limit. If the card is ever compromised, you cancel it with one click. The AI can't accidentally (or maliciously) run up charges.

Scoped tokens always. Every API token should have the minimum permissions needed. Read-only where possible. Repo-specific where possible. Short expiration where possible.

The test: If the AI's entire identity were compromised tomorrow, what's the worst that happens? The answer should be "we lose a throwaway Gmail and a $20 Privacy.com card." If the answer involves your personal accounts, bank, or employer ‚Äî you've set it up wrong.
4

Phase 4: SOUL.md ‚Äî Behavioral Guardrails

SOUL.md is the AI's constitution. Put safety rules at the very top ‚Äî before personality, before instructions, before anything else. Here's a template:

SOUL.md safety rules template:
## Safety Rules (non-negotiable) ### File Access - NEVER read/write outside workspace without explicit approval - NEVER access ~/.ssh, ~/.aws, ~/.config, /etc, or system files - NEVER modify dotfiles (.bashrc, .zshrc, .gitconfig, etc.) ### Command Execution - NEVER run: rm -rf, sudo, chmod 777, or recursive deletes - NEVER install system packages without approval - NEVER make outbound network calls to unknown endpoints - NEVER execute downloaded/piped scripts without review ### Credentials - NEVER log, echo, or store tokens/passwords in plain text - NEVER commit secrets to git - NEVER share credentials in chat messages - If you see a secret in output, redact it immediately ### Communication - NEVER send emails, tweets, or public messages without approval - NEVER impersonate the human - NEVER share private information in group chats ### Spending - NEVER make purchases over $10 without explicit approval - NEVER sign up for paid plans without approval - NEVER change billing or subscription settings without approval
Important: SOUL.md is guidance, not enforcement. It's like a company policy ‚Äî it works because the AI follows it, not because it's technically impossible to violate. Real enforcement comes from sandboxing, file system restrictions, and network controls (see Phase 5).
5

Phase 5: AGENTS.md ‚Äî Operational Rules

AGENTS.md controls execution policy. Set it to require approval for anything risky:

Recommended execution policy:
host: sandbox security: allowlist ask: on-miss

High-risk operations should always require manual approval:

‚Ä¢ rm, recursive deletes, sudo, chmod
‚Ä¢ curl, wget, outbound network calls
‚Ä¢ Writes to config, SSH, AWS, or env paths
‚Ä¢ npm install, pip install, brew install
‚Ä¢ git push to main/master

Lock the filesystem:

Restrict file access to workspace only:
tools.fs.workspaceOnly: true tools.exec.applyPatch.workspaceOnly: true
6

Phase 6: Channel Security

DM pairing / allowlists. Only your account should be able to talk to the bot. Never set dmPolicy: "open" ‚Äî that lets anyone message it and give it instructions.

Group chats: require @mention. Set requireMention: true so the bot doesn't respond to every message. Other group members could craft messages that trick the AI (prompt injection). If it only responds when mentioned, the attack surface shrinks dramatically.

Consider a separate phone number (eSIM or cheap prepaid) for any messaging apps that require one.

7

Phase 7: Ongoing Hygiene

Update frequently.

Keep OpenClaw up to date:
brew upgrade openclaw && openclaw security audit --deep

Review logs weekly for unexpected commands, outbound network calls, or file access outside the workspace.

Never install skills from untrusted sources. Community skill/soul packs can contain hidden prompt injections. Treat downloaded .md files as untrusted code ‚Äî because they are.

The Billboard Test: "Would I be comfortable with this on a billboard?" If the answer is no, don't share it with the AI.

‚ö†Ô∏è What's Still Risky Even With All This

1. Prompt injection is unsolved. A malicious email to the AI's Gmail could trick it into taking actions you didn't authorize. That's why it gets a throwaway email with no sensitive data.

2. The LLM provider sees everything. Your prompts, the AI's responses, file contents it reads ‚Äî all of it passes through the API. Don't feed it anything truly secret.

3. Skills can be malicious. Only use skills you've reviewed yourself. A skill is just a markdown file that can tell the AI to do anything.

4. SOUL.md is not a firewall. It's guidance, not enforcement. Real enforcement = sandboxing, file system restrictions, network controls.
